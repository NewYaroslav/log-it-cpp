# Playbook по владению заголовками и файлами реализации

Используйте этот документ, когда меняете структуру `.hpp`, `.ipp` или `.tpp`.

Канонической версией является английская:

- `agents/header-implementation-guidelines.md`

## Сначала выберите модель подсистемы

Перед правками заголовков определите один основной подход:

- **aggregate-first**
  - обычный consumer сначала подключает один umbrella header
  - общие prerequisites можно централизовать там
  - leaf headers могут опираться на уже подготовленный include-context
- **standalone-leaf-first**
  - leaf headers должны быть самодостаточными
  - каждый leaf сам владеет зависимостями своего публичного контракта

Не смешивайте эти модели внутри одной небольшой подсистемы без необходимости.

Для `log-it-cpp` публичные модули в основном используют aggregate-first:

- сначала подключайте `<logit.hpp>`, `<logit/utils.hpp>`,
  `<logit/formatter.hpp>` или `<logit/loggers.hpp>`
- не обходите этот контракт прямыми include соседних модулей

## Раскладывайте код по ownership

### `.hpp`

В `.hpp` должны жить:

- публичные объявления
- публичные alias-типы и config-структуры
- include-контракт модуля

### `.tpp`

В `.tpp` должна жить:

- template-visible реализация

Правила:

- если consumer может инстанцировать шаблон, definition должна быть видна из заголовка
- `.tpp` при необходимости подключается из `.hpp` безусловно
- нельзя прятать template definitions за implementation-only путями

### `.ipp`

В `.ipp` должна жить:

- нетemplate-реализация, которую удобно держать рядом с заголовком
- общая реализация для header-only и compiled-library режимов
- inline-capable helper bodies

Для `.ipp` заранее решите, является ли он:

- всегда header-owned
- или ownership-controlled через compiled ownership TU

## Делайте ownership явным

Always-included реализацию используйте только для:

- шаблонов
- маленьких helper-ов
- коротких inline-методов
- логики, которая действительно принадлежит заголовку

Ownership-controlled реализация нужна для:

- больших нетemplate-методов
- кода, который должен эмититься один раз в compiled-library режиме
- общей реализации для header-only и compiled-library сборок

## Размещайте зависимости осознанно

Централизуйте общие include только тогда, когда подсистема реально строится
вокруг umbrella header.

Иначе:

- держите зависимости рядом с leaf header, который владеет публичным контрактом

Implementation-only helper-ы вроде build-mode macros держите на самой узкой
корректной границе:

- предпочтительно в `.ipp`, который их использует
- иначе в непосредственном owning `.hpp`

Не раздавайте такие helper-ы через широкие агрегаторы без реальной необходимости.

## Соблюдайте границы include

- Не используйте `../` в `#include`.
- Внутри одной модульной ветки подключайте локальные заголовки forward-путями,
  например `#include "compiler/PatternCompiler.hpp"`.
- Межмодульные зависимости должны приходить через ближайший umbrella header,
  а не через прямые include соседних модулей.
- Файлы под `logit/detail/` могут подключать другие detail-заголовки, но не
  публичные `logit/...` headers.
- Consumers не должны подключать `logit/detail/...` напрямую.

## Осторожно с forward declarations

Добавляйте forward declarations только тогда, когда они действительно дают
структурную пользу:

- разрывают цикл
- уменьшают неизбежную связанность
- стабилизируют настоящий leaf interface

Не добавляйте их только ради визуальной "легкости", если это:

- скрывает зависимости
- повышает чувствительность к include-order
- ухудшает читаемость подсистемы

Если подсистема aggregate-first, обычно лучше сохранить ясную и плоскую
структуру include-зависимостей, чем механически плодить forward declarations.

## Соблюдайте private boundaries

Не подключайте private или vendor `.ipp` из пользовательского кода ради обхода
public API.

Если публичных заголовков не хватает:

- либо корректно расширьте зависимость
- либо держите нужный ownership локально

## Тестируйте intended contract

Для aggregate-first подсистем:

- тестируйте публичный путь через umbrella header
- ориентируйтесь на include-проверки в стиле NHR, уже используемые в репозитории

Для standalone-leaf-first подсистем:

- тестируйте прямое подключение leaf headers там, где это часть контракта

Для white-box тестов:

- прямое подключение leaf headers допустимо
- тест должен сам явно подключать свои prerequisites
- internal tests не должны случайно переопределять публичную include policy

## Быстрый checklist

Перед финализацией проверьте:

- модель подсистемы выбрана явно
- `.tpp` содержит все template-visible definitions, нужные consumer-ам
- ownership `.ipp` задан явно и последовательно
- общие include централизованы только там, где aggregate model действительно осознанна
- implementation-only helper-ы остаются локальными
- forward declarations действительно оправданы
- тесты проверяют именно intended include contract
